# CODEX_KIMI_TASK_ALLOCATION

Author: CODEX
Date: 2026-04-26
Status: Vigente

## Objetivo

Definir distribuicao permanente de trabalho entre CODEX e KIMI para maximizar velocidade, qualidade e chance de sucesso, sem overlap inutil.

## Principio de sucesso

A melhor divisao nao e a mais simetrica.

A melhor divisao e a que:

1. reduz conflito de escrita
2. preserva fonte da verdade
3. acelera descoberta e consolidacao
4. usa cada agente onde ele gera mais ganho liquido

## Donos preferenciais

### CODEX deve liderar

1. runtime critico do brain
2. hardening estrutural
3. integridade de sync, snapshot, rollback e dashboard
4. consolidacao de verdade operacional
5. revisao final de regressao
6. promocoes para guardrails e patterns

### KIMI deve liderar

1. exploracao ampla de gaps e oportunidades
2. pesquisa estrategica e varredura de backlog
3. propostas de evolucao do sistema
4. estrutura de learning system e memoria pesquisavel
5. consolidacao de personalidades e documentacao de uso
6. descoberta de ideias, heuristicas e automacoes de alto alcance

### Requer consenso CODEX + KIMI

1. mudancas em runtime critico do brain principal
2. novas politicas de swarm/subagentes
3. alteracoes de source-of-truth
4. score model e metricas de maturidade
5. qualquer mudanca que altere governanca entre agentes

## Anti-overlap

### KIMI nao deve puxar para si sem consenso

1. lock strategy do sync
2. verdade final de dashboard vs snapshot
3. autoria compartilhada em metadata critica
4. narrativa de runtime quando a execucao mudou

### CODEX nao deve puxar para si sem necessidade

1. exploracao ampla de oportunidades de backlog
2. brainstorm estrategico de features futuras
3. catalogacao inicial de learning system
4. discovery paralelo quando a pergunta ainda esta aberta

## Cadencia de handoff

### Loop ideal

1. KIMI explora e propõe
2. CODEX audita e endurece
3. KIMI expande em cima da fundacao validada
4. CODEX consolida status vigente e aprendizado

## Regra de execucao por tipo de trabalho

### Tipo A - Exploracao
- lead: KIMI
- review: CODEX

### Tipo B - Hardening
- lead: CODEX
- support: KIMI

### Tipo C - Sistema novo
- KIMI propõe arquitetura e learning surface
- CODEX define guardrails, ownership e criterio de promocao

### Tipo D - Divergencia
- ambos produzem relato curto
- vence evidencia live, reproduzivel e mais recente

## Regra permanente

Sempre que houver novo plano em `decisions.md`:

1. identificar o que e discovery
2. identificar o que e hardening
3. atribuir discovery preferencialmente para KIMI
4. atribuir hardening preferencialmente para CODEX
5. registrar a divisao antes de executar muito

## Modo budget-aware

A distribuicao anterior e preferencial, nao limitadora.

### Regra nova

1. CODEX e KIMI podem fazer qualquer tipo de trabalho quando necessario.
2. ownership padrao continua valendo para reduzir conflito.
3. quando houver vantagem clara de credito, throughput ou exploracao paralela, KIMI pode assumir mais frentes, inclusive execucao ampla.
4. CODEX continua como consolidador preferencial de verdade operacional em runtime critico, salvo indisponibilidade ou custo proibitivo.

### Politica de uso de credito

- usar KIMI como motor de volume quando a tarefa exigir:
  1. pesquisa extensa
  2. exploracao paralela
  3. varredura de backlog
  4. gerações amplas de proposta
  5. trabalho iterativo de alto consumo

- usar CODEX como motor de precisao quando a tarefa exigir:
  1. hardening fino
  2. validacao live
  3. consolidacao final
  4. revisao de regressao
  5. fechamento de runtime critico

### Regra mestra revisada

Ambos fazem tudo quando necessario.
A diferenca e de alocacao otima, nao de capacidade.
